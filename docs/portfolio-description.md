# MarketPulse — 포트폴리오 설명

## 프로젝트 요약

CBE(Cross-Border Ecommerce) 도메인을 경험하며 각 국가별, 채널별 다양한 화주의 주문 데이터가 단순 운영에만 쓰이는 것이 아깝다는 생각에서 시작했습니다. 뷰티/화장품 상품의 비정형 속성을 구조화하고, Neo4j 인과 체인 위에서 LLM이 데이터 소스를 자율 조합해 셀러에게 국가별 시장 인사이트를 제공하는 멀티에이전트 시스템입니다. 데이터에서 발견된 통계적 패턴이 사용자 승인을 거쳐 온톨로지에 편입되고, 이후 답변에 실제 반영되는 — 쓸수록 지식 기반이 성장하는 구조를 설계·구현했습니다.

---

### | 맥락 의존적 속성 추출을 위한 Tool Use + Dynamic Few-Shot 파이프라인

주문 데이터에서 국가별·채널별 인사이트를 뽑으려면, 비정형 상품명에서 분석 가능한 속성을 먼저 구조화해야 합니다. 문제는 뷰티 속성이 맥락 의존적이라는 점입니다. "무기자차"는 선크림에선 UV 차단 방식이지만 토너에선 아무 의미가 없고, "77"이 성분 함량인지 모델 번호인지는 카테고리마다 다릅니다. 키워드 매칭이나 룰로는 근본적으로 풀 수 없는 문제라 LLM 기반 접근을 택했는데, LLM만 쓰면 대량 추출 시 출력 형식이 들쑥날쑥해서 후속 집계가 깨집니다. 그래서 Tool Use로 출력 구조 자체를 강제하고, Dynamic Few-Shot으로 값 표기 방식까지 유도하는 2-레이어 구조로 설계했습니다. Few-Shot 예시는 검수된 정답 데이터를 벡터 DB에 임베딩해두고, 입력 상품명과 유사도 검색으로 가장 적합한 것을 골라 LLM에 넣어줍니다. LangChain 없이 Anthropic SDK를 직접 쓴 건, 도구 호출 로깅이나 프롬프트 버전 관리, 검증 흐름 같은 것들을 프레임워크한테 맡기면 제어가 안 되기 때문입니다.

### | 인과 추론과 통계 집계를 LLM이 자율 조합하여 인사이트를 도출하는 에이전트 설계

SQL로 "일본 선크림 중 비건 58%"는 뽑을 수 있는데, "왜 58%인가?"는 SQL만으로 답이 안 됩니다. Neo4j에 기후→피부고민→기능→성분의 인과 체인을 온톨로지로 구축하고, LLM이 ReAct 패턴으로 Neo4j(인과 근거)와 PostgreSQL(수치 집계)을 알아서 조합하게 만들었습니다. 도구 스키마는 Pydantic 모델에서 자동 생성되니까 도구를 추가해도 서버 파일 하나만 건드리면 되고, 매 도구 호출마다 "왜 이 도구를 골랐는지"(reasoning), "뭘 호출했는지"(decision), "결과가 뭐였는지"(result)를 전부 trace로 남겨서 LLM의 판단 과정을 투명하게 추적할 수 있습니다.

### | PatternScout: 데이터에서 통계적 패턴을 탐지하여 온톨로지를 지속 확장

도메인 전문가가 미리 정의한 인과 체인만으로는 데이터에서 새로 생기는 속성 간 관계를 못 잡습니다. 시스템이 쓰일수록 점진적으로 더 촘촘한 관계망을 갖도록, 주문 데이터에서 통계적 공존 패턴을 자동 탐지하는 별도 에이전트를 만들었습니다. LLM한테는 "어떤 속성 쌍을 검사할지" 판단만 시키고, 동시 출현 빈도나 시계열 상관 같은 수치는 SQL과 통계 라이브러리가 돌아가는 전용 도구로만 나오게 해서, LLM이 근거 없는 숫자를 만들어내는 걸 구조적으로 막았습니다. 탐지된 패턴은 사용자가 직접 검토하고 승인해야 온톨로지에 들어가며, 승인 후 같은 질문을 다시 던지면 AnalystAgent가 통계적 근거를 인용해서 답변하는 것을 Before/After로 정량 확인했습니다.

### | 인과관계 확장이 질의에 미치는 영향을 추적하는 Eval 설계

관계를 새로 승인하거나, 프롬프트를 고치거나, 데이터가 추가됐을 때, 그게 최종 답변을 진짜 좋게 만들었는지 숫자로 볼 수 있어야 합니다. 패턴 탐지 현황, 답변 내 승인된 관계 활용률, 인과 근거 커버리지, 비용 효율 4축으로 시스템 성장을 측정하는 Eval을 짰고, 변경 전후 스냅샷을 자동 비교해서 어떤 축이 얼마나 움직였는지 확인할 수 있습니다. 안 쓰기로 한 지표(LLM이 LLM을 채점하는 방식, 답변 길이 비교 같은 것)와 왜 안 썼는지도 ADR로 남겨서, 뭘 측정 안 하기로 한 것도 설계 결정으로 관리하고 있습니다.

### 주요 구현 내용

- 비정형 상품명에서 맥락 의존적 속성을 구조화하는 Few-Shot + Tool Use 추출 파이프라인 구현
- LLM이 Neo4j(인과) + PostgreSQL(집계) 도구를 자율 조합하는 ReAct 오케스트레이터 구현
- Pydantic 기반 도구 스키마 자동 생성 구조 설계 (도구 추가 시 단일 파일 수정)
- 통계적 패턴을 코드가 계산하고 LLM이 제안하는 PatternScout 에이전트 구현
- 제안→사용자 승인→온톨로지 편입→답변 변화의 Human-in-the-loop 지식 확장 파이프라인 구현
- 4축 Eval 프레임워크 + 변경 전후 스냅샷 비교 시스템 구현
- 의사결정 과정을 Think/Act/Observe로 시각화하는 단일 페이지 대시보드 구현
- 19개 Architecture Decision Record로 설계 결정의 맥락·대안·근거를 문서화
- 69건 단위 테스트 + 4건 실제 Claude API 통합 테스트

### 기술 스택

Python, FastAPI, Claude API (Sonnet 4 / Tool Use), Neo4j, PostgreSQL, ChromaDB, scipy, Next.js, React, shadcn/ui, Recharts

# MarketPulse — 포트폴리오 설명

## 프로젝트 요약

Cross-Border Ecommerce 도메인에서 실 주문 데이터가 운영 목적으로만 쓰이는 것에 한계를 느끼며 고안한, 뷰티/화장품 상품의 비정형 속성을 구조화하고 국가별 시장 인사이트를 자율 생성하는 LLM 멀티에이전트 시스템입니다. 상품명의 속성이 맥락 의존적이라 룰 기반으로 해결할 수 없는 문제를 정의하고, Tool Use + Dynamic Few-Shot 기반 추출 파이프라인, Neo4j 인과 체인 위에서 LLM이 데이터 소스를 자율 조합하는 ReAct 오케스트레이터, 그리고 데이터에서 통계적 패턴을 탐지하여 온톨로지를 지속 확장하는 PatternScout 에이전트까지, 시스템이 사용될수록 지식 기반이 성장하는 구조를 설계·구현했습니다.

---

### | 맥락 의존적 속성 추출을 위한 Tool Use + Dynamic Few-Shot 파이프라인

주문 데이터에서 국가별·채널별 인사이트를 도출하려면, 비정형 상품명에서 분석 가능한 구조화된 속성을 먼저 추출해야 합니다. 역직구 뷰티 상품명에서 "무기자차"는 선크림에선 UV 차단 방식이지만 토너에선 무의미하고, "77"이 성분 함량인지 모델 번호인지는 카테고리에 따라 달라집니다. 이 맥락 의존성 때문에 키워드 매칭이나 룰 기반 분류가 원천적으로 불가능하다는 것을 문제로 정의하고, LLM 기반 접근이 본질적 해결책임을 도출했습니다. 단, LLM 단독으로는 대량 추출 시 출력 형식이 일관되지 않아 후속 집계가 불가능하므로, Tool Use로 출력 구조를 강제하고, Dynamic Few-Shot으로 값 표기 형식을 유도하는 2-레이어 전략을 설계했습니다. Few-Shot에 주입할 예시는 사전 검수된 정답 데이터를 ChromaDB에 임베딩해두고, 입력 상품명과의 벡터 유사도 검색으로 가장 유사한 예시를 동적으로 선택하여 LLM에 주입합니다. LangChain을 사용하지 않은 것은 의도적 선택으로, 도구 호출 로깅, 프롬프트 버전 관리, 검증 파이프라인의 세밀한 제어가 프레임워크 추상화 뒤에 숨으면 불가능해지기 때문입니다.

### | 데이터 소스를 LLM이 자율 조합하는 ReAct 오케스트레이터 설계

SQL 집계로는 "비건 58%"라는 숫자만 나오고, "왜 58%인가?"에는 답할 수 없습니다. 이를 해결하기 위해 Neo4j에 기후→피부고민→기능→성분의 인과 체인을 온톨로지로 구축하고, LLM이 while-loop ReAct 패턴으로 Neo4j(인과)와 PostgreSQL(집계)을 자율 조합하여 인사이트를 생성하도록 설계했습니다. 도구 스키마를 Pydantic BaseModel + @tool 데코레이터로 자동 생성하여 도구 추가 시 서버 파일 1곳만 수정하면 되는 구조를 적용했으며, 모든 도구 호출의 reasoning(사고) + decision(도구 선택) + result(결과)를 trace에 기록하여 LLM의 판단 과정을 투명하게 추적할 수 있게 했습니다.

### | PatternScout: 데이터에서 통계적 패턴을 탐지하여 온톨로지를 지속 확장

시드 온톨로지가 정적이라 데이터에서 새롭게 발생하는 패턴을 반영하지 못하는 한계를 해결하기 위해, 주문 데이터에서 통계적 공존 패턴을 자동 탐지하는 별도 에이전트를 설계했습니다. LLM에게는 통계 도구(compute_cooccurrence_lift 등)를 호출할 파라미터 선택만 맡기고, lift·correlation·p-value 계산 자체는 SQL 집계와 scipy가 수행하는 전용 도구로 분리하여, LLM이 직접 숫자를 생성할 수 없는 구조를 만들었습니다. 승인된 관계는 Neo4j에 편입되어 이후 AnalystAgent 답변에 실제 반영되며, Before/After 비교에서 같은 질문의 답변이 "각각 따로 설명"에서 "lift=1.55의 통계적 시너지 관계"로 변화하는 것을 정량적으로 확인했습니다.

### | 변경 영향도 측정을 위한 Eval 설계

PatternScout가 제안한 관계를 승인했을 때, 그것이 실제 답변 품질을 개선하는지를 정량적으로 판단할 수 있어야 합니다. 이를 위해 4축 Eval(패턴 탐지 현황 / 답변 품질 개선률 / 추론 커버리지 / 시스템 효율)을 설계하고, 핵심 지표인 discovered_usage_rate(승인된 관계의 실제 답변 활용률)가 Before 0%에서 After 20%로 변화하는 것을 스냅샷으로 정량화했습니다. 기각한 지표(LLM-as-judge, 답변 길이, evidence density)와 그 이유를 ADR로 문서화하여, "무엇을 측정하지 않기로 했는가"도 설계 결정으로 관리했습니다.

### 주요 구현 내용

- 비정형 상품명에서 맥락 의존적 속성을 구조화하는 Few-Shot + Tool Use 추출 파이프라인 구현
- LLM이 Neo4j(인과) + PostgreSQL(집계) 도구를 자율 조합하는 ReAct 오케스트레이터 구현
- Pydantic @tool 데코레이터로 도구 스키마를 자동 생성하는 구조 설계
- 통계적 패턴을 코드가 계산하고 LLM이 제안하는 PatternScout 에이전트 구현
- 제안→승인→온톨로지 편입→답변 변화의 Human-in-the-loop 지식 확장 파이프라인 구현
- 4축 Eval 프레임워크 + Before/After 스냅샷 비교 시스템 구현
- 의사결정 과정을 Think/Act/Observe로 시각화하는 단일 페이지 대시보드 구현
- 19개 Architecture Decision Record로 설계 결정의 맥락·대안·근거를 문서화
- 69건 단위 테스트 + 4건 실제 Claude API 통합 테스트

### 기술 스택

Python, FastAPI, Claude API (Sonnet 4 / Tool Use), Neo4j, PostgreSQL, ChromaDB, scipy, Next.js, React, shadcn/ui, Recharts

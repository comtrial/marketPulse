# ADR-011: 추출 단계에서 원본 키워드 보존 원칙

- **상태**: accepted
- **날짜**: 2026-03-28

## 맥락

bootstrap_extract.py에서 "톤업"을 "브라이트닝"으로, "노세범"을 "피지조절"로 동의어 변환하여 `functionalClaims`에 저장했다. 이로 인해 `get_attribute_trend("톤업")`으로 검색하면 0건이 반환되어, 패턴 C(JP 톤업 67%→43% 하락)를 검증할 수 없었다.

## 결정

**추출 단계에서는 상품명의 키워드를 그대로 저장한다.** 동의어 변환은 추출 단계에서 하지 않는다.

```
추출 단계: "톤업" → functionalClaims: ["톤업"]  (그대로 저장)
집계 단계: "톤업 ∈ 브라이트닝" 매핑이 필요하면 별도 레이어에서 처리
```

## 근거

1. **검색 가능성**: "톤업"으로 검색하면 "톤업"이 나와야 한다. "브라이트닝"으로 변환하면 원본 키워드 검색이 불가.

2. **정보 손실 방지**: "톤업"과 "미백"은 둘 다 브라이트닝 카테고리지만 의미가 다르다. "톤업 = 즉각적 톤 보정", "미백 = 장기적 색소 억제". 같은 상위 카테고리로 합치면 이 구분이 사라진다.

3. **LLM 추출과의 일관성**: 실제 LLM(Claude) 추출에서도 "톤업"은 "톤업"으로 나올 것이다. Tool schema의 description에 "톤업, 노세범, 수분..." 등 원본 키워드를 예시로 제공하기 때문. bootstrap 추출과 LLM 추출의 출력이 같은 형식이어야 집계가 깨지지 않는다.

4. **집계 유연성**: 원본을 보존하면 나중에 다양한 기준으로 집계 가능:
   - 원본 그대로: "톤업 58%, 미백 12%"
   - 상위 카테고리: "브라이트닝(톤업+미백) 70%"
   - 역변환은 불가: "브라이트닝 70%" → 톤업이 몇 %인지 복원 불가

## 적용 범위

- `bootstrap_extract.py`: FUNC_PATTERNS에서 동의어 변환 제거
- `extraction/tool_schema.py`: functionalClaims description에 원본 키워드 예시 ("톤업, 노세범, 수분, 진정...")
- 성분 키워드도 분리: "히알루론산"은 keyIngredients에만, functionalClaims("수분공급")에 혼입하지 않음

## 결과

- **장점**: 패턴 A/B/C 검증 정상 (19%→56%, 71~80%, 67%→44%)
- **장점**: LLM 추출과 bootstrap 추출의 출력 형식 일관
- **장점**: 상위 카테고리 집계를 Phase 2에서 유연하게 설계 가능
- **후속**: Neo4j Function 노드에 aliases 속성 추가 또는 ALIAS_OF 관계 (Phase 2)

# ADR-001: Neo4j 온톨로지 기반 지식 그래프

- **상태**: accepted
- **날짜**: 2026-03-27

## 맥락

뷰티/화장품 도메인에서 단순 SQL 집계로는 "일본에서 비건 비율이 58%"라는 숫자만 나온다.
실제 필요한 것은 **"일본 여름 고UV → UV손상 우려 → 클린뷰티 트렌드 → 비건 선크림 수요 급증"** 같은 인과 체인 도출이다.

또한 뷰티 속성은 하위 카테고리마다 의미가 다르고(선크림의 "무기자차" ≠ 토너), 신규 트렌드("마이크로바이옴")가 계속 등장하므로 고정 스키마로는 도메인 지식을 표현할 수 없다.

## 결정

**Neo4j 그래프 DB**를 사용하여 온톨로지 기반 지식 그래프를 구축한다.

- 노드: Ingredient, Attribute, Category, Country, Season, Trend 등
- 관계: `HAS_ATTRIBUTE`, `CAUSES`, `POPULAR_IN`, `SEASONAL_TREND` 등
- 추출된 성분명은 Neo4j의 Ingredient 노드와 MERGE되어 인과 체인에 연결

## 근거

| 방식 | 인과 체인 표현 | 스키마 유연성 | 탐색 쿼리 성능 |
|------|--------------|-------------|--------------|
| **Neo4j (선택)** | 네이티브 관계 탐색 | 스키마리스, 노드/관계 자유 추가 | 관계 탐색 O(1) hop |
| PostgreSQL JSON | JSON 중첩으로 표현 가능하나 복잡 | JSON이라 유연하지만 쿼리 어려움 | JOIN 폭발 |
| MongoDB | 문서 중첩 | 유연하나 관계 표현 약함 | 관계 탐색에 부적합 |

도메인 지식을 "정의"하고 확장해갈 수 있는 구조가 필요했고, 온톨로지 방식이 이에 가장 적합하다.

## 결과

- **장점**: 인과 체인 쿼리가 Cypher로 직관적, 도메인 전문가가 관계를 직접 정의/확장 가능
- **장점**: LLM이 그래프를 도구로 탐색하며 추론 근거를 명시적으로 제시 가능
- **단점**: Neo4j 운영 복잡도 추가 (클러스터링, 백업)
- **단점**: PostgreSQL과 이중 관리 필요 → 역할 분리를 명확히 해야 함
- **후속**: Neo4j 노드/관계 스키마 상세 설계, Docker Compose에 Neo4j 추가
